Scan

Hardware + Software Detection

  • RAM via Win32_ComputerSystem (TotalPhysicalMemory)
  • GPU via nvidia-smi (accurate VRAM), WMI fallback
  • Disk free space on target drive (D: if exists, else C:)
  • Local AI verdict: READY / POSSIBLE / NOT RECOMMENDED / NO GPU
  • Internet speed via Cloudflare 5 MB download
  • Scan 20 items in software catalog (winget, npm, ollama)
  • Report: X installed, Y available
Could improve
  • WMI AdapterRAM DWORD cap: done, falls back to registry qwMemorySize (64-bit)
  • Cross-platform: done, startupjet.sh detects macOS (system_profiler) and Linux (nvidia-smi/lspci)
  • AMD ROCm detection: could probe rocminfo or hipconfig to confirm ROCm works
  • Multi-GPU: only picks the best GPU, does not report if a second GPU exists
2
Choose

All Questions Upfront

  • Preset profiles: Minimal dev, Developer, AI workstation, or Custom
  • Model recommendation engine: score all models by VRAM, RAM, disk, show top 3
  • If Custom: install mode [1] recommended, [2] skip AI, [3] customize, [4] skip, [5] all
  • Install scope: current user vs all users (winget --scope)
  • Auth choices: GitHub CLI, Tailscale, cloudflared
  • Generate SSH key? (if none exists at ~/.ssh/id_ed25519)
  • Apply Windows dev settings? (file extensions, hidden files, dev mode)
  • Install VS Code extensions? (from config/vscode-extensions.json)
  • Workspace path, GitHub username, git email (with defaults from config/defaults.json)
  • Time estimate based on selections + measured download speed
Could improve
  • Preset profiles: implemented (Minimal dev, Developer, AI workstation). Reduces Phase 2 to one question.
  • A config/presets.json file so users can define their own presets
  • Model recommendations could show estimated inference speed (tokens/sec) per model/hardware combo
  • Show disk space impact of each selection in real-time as user picks
3
Auth

Interactive Authentication

  • If gh not yet installed but selected: defer auth to after install
  • gh auth login (opens browser)
  • tailscale up (opens browser, if chosen)
  • cloudflared tunnel login (opens browser, if chosen)
Could improve
  • Auth verification after each login (currently only gh gets checked via auth status)
  • Tailscale and cloudflared auth status checks before prompting (avoid re-auth if already logged in)
  • Admin elevation now offered when machine scope or dev mode is selected (implemented)
4
Configure

Git, SSH, Dev Settings, Config Save

  • git config (user.name, user.email) with --global or --system per scope
  • ssh-keygen -t ed25519 (if chosen and key missing)
  • gh ssh-key add (auto-upload to GitHub if gh authenticated)
  • Registry edits: HideFileExt=0, Hidden=1, DevMode (needs admin)
  • Save all choices to config/user-config.json
Could improve
  • Vault integration: done, Bitwarden CLI restores SSH keys from vault before generating new ones
  • Admin check: done, detects privileges before attempting dev mode registry writes
  • Back up existing git config before overwriting (in case user has custom aliases, etc.)
  • SSH config file (~/.ssh/config) auto-generation for common hosts
  • GPG key setup alongside SSH key
NO MORE USER INPUT AFTER THIS POINT
5
Install

Unattended Installation

  • 5a: winget install (Git, gh, Python, pwsh, VS Code, Tailscale, cloudflared, Node, Ollama, uv)
  • 5b: manual installs (OpenSSH via Add-WindowsCapability)
  • 5c: npm install -g (Claude Code, OpenAI Codex)
  • 5d: ollama pull (recommended or all selected models)
  • 5e: deferred auth (gh, tailscale, cloudflared installed above)
  • 5f: deferred git config (if git was just installed)
  • 5g: deferred SSH key add to GitHub
  • 5h: VS Code extensions via code --install-extension
  • Progress saved to config/progress.json after each item
  • Refresh-SessionPath after each group (picks up new PATH entries)
Could improve
  • Parallel installs: winget items could run concurrently (winget supports it but may conflict)
  • Retry logic: if a winget install fails, retry once before marking as failed
  • Model pull progress: ollama pull shows progress but it is captured in transcript, not displayed live
  • Validate installed version matches expected minimum (e.g. Python 3.10+)
  • Post-install PATH validation: check that each tool's binary is actually reachable after Refresh-SessionPath
5.5
Dotfiles

Dotfiles Clone + Link

  • Read config/dotfiles.json for repo URL and file mappings
  • If repo is empty string: skip entire phase
  • git clone dotfiles repo to ~/.dotfiles (or git pull if exists)
  • Windows: copy files to target paths (with .backup of existing)
  • macOS/Linux: symlink files to target paths (with .backup of existing)
Could improve
  • Stow-compatible layout for users who already use GNU Stow
  • Per-OS file mappings (e.g. different shell configs for macOS vs Linux)
6
Clone

Clone Repos from Config

  • Read config/repos.json for repo list
  • If repo dir already exists: skip
  • git clone each repo to <workspace>/github/<name>
  • If clone fails + required=true: report FAIL. If optional: skip with warning
  • Shallow clone: repos with "shallow": true use --depth 1
  • Post-clone deps: detects package.json / requirements.txt, runs npm install / pip install
Could improve
  • SSH vs HTTPS clone: detect if SSH key exists + gh auth, prefer SSH for push access
  • Branch checkout: allow repos.json to specify a branch to checkout after clone
7
Verify

PATH Check + Functional Tests + Summary

  • Check every tool is on PATH with --version output
  • SSH key existence check
  • Smoke test: write + delete temp file in ai-journal
  • gh api user to confirm authenticated identity
  • Functional: Python import test (json, os, sys)
  • Functional: Node.js eval test
  • Functional: Ollama inference test with response timing
  • Functional: git config identity check
  • Summary: installed, already had, failed, authenticated, repos, time elapsed
  • Clean up progress.json if no failures (or keep for resume)
  • Print canonical AI prompt for next step
Could improve
  • Functional tests: done, Python import test, Node.js test, Ollama inference with timing, git identity check
  • Smoke test could try git push (not just write) to confirm full auth flow
  • Generate a machine profile JSON (hardware + software + config) for future reference
  • Email or webhook notification when setup completes (useful for unattended runs)

Data flow between phases

Variable Set in Used in Purpose
$vramGB Phase 1 Phase 1 (verdict), Phase 2 (recommendations) GPU VRAM in GB, drives model selection
$ramGB Phase 1 Phase 2 (recommendations) System RAM in GB, filters large MoE models
$freeGB Phase 1 Phase 2 (recommendations) Disk free space, filters models by download size
$downloadMBps Phase 1 Phase 2 (time estimate) Measured download speed for ETA calculation
$localAiCapable Phase 1 Phase 2 (UI, warnings, recommendations) Boolean: can this PC run local AI at all?
$recommendedModels Phase 2 Phase 2 (option [1], [3] listing) Top 3 models ranked by hardware fit
$catalog[].selected Phase 2 Phase 5 (install loop) Which items the user chose to install
$installScope Phase 2 Phase 4 (git config), Phase 5 (winget --scope) "user" or "machine"
$workspacePath Phase 2 Phase 4 (config save), Phase 6 (clone target) Root workspace directory
config/progress.json Phase 5 Phase 1 (resume check), Phase 7 (cleanup) Tracks completed installs for crash resume
